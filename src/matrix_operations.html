Matrix Operations
GPU Training vs CPU Inference

// making the math concrete

GPU

Training — Massive Parallel Matrix Multiplication

During training, the model processes an entire batch of examples at once. For each layer, it computes Output = Input × Weights — but the Input is not one vector, it's a whole matrix (one row per sample in the batch). Every cell in the output matrix is an independent dot product — perfect for GPU parallelism.

// training also does backpropagation — even more parallel matrix ops
Forward pass
Output = Input × W     [batch × out_dim]
∝ batch
Loss grad
dL/dOutput   [batch × out_dim]
∝ batch
Weight grad
dL/dW = Inputᵀ × dL/dOutput   [in_dim × out_dim]
parallel ✓
Input grad
dL/dInput = dL/dOutput × Wᵀ   [batch × in_dim]
parallel ✓
Why GPU wins at training: Every cell of every matrix can be computed independently and simultaneously. A batch of 32 samples × a layer of 4096 neurons = 131,072 dot products — all at once. The GPU's thousands of cores are fully occupied. No core waits for another.
CPU

Inference — Vector × Matrix, One Token at a Time

During inference, there is no batch — just one token. So the "Input" shrinks from a fat matrix down to a single row vector. The operation becomes vector × matrix (not matrix × matrix). This means far fewer operations, and critically, far less parallelism available.

▶ CPU: Vector × Matrix — One Token Inference (click "Animate")
// inference is sequential — what one CPU core does per token
Step 1 — memory
LOAD weight column 0 from RAM → cache
Must read billions of floats for the full model. Dominates total time.
⚠ expensive
Step 2 — compute
dot(token_vec, weight_col_0) → output[0]
Tiny multiply-accumulate loop. Finishes in nanoseconds.
✓ cheap
Step 3 — memory
LOAD weight column 1 from RAM → cache
Next column. Memory stall again.
⚠ expensive
Step 4 — compute
dot(token_vec, weight_col_1) → output[1]
Again, instant math after slow memory load.
✓ cheap
… repeat for all columns …
LOAD → DOT → LOAD → DOT → …
The pattern: 95% memory, 5% math. CPU handles this naturally.
…
Why CPU is suitable for inference: With only one token (one vector), there's no big matrix to parallelize over. The operation is dominated by loading weights from memory, not by math. CPUs stream memory efficiently, have large L3 caches, and can access cheap abundant RAM — perfectly matched to this "read a lot, compute a little" pattern. GPU cores would sit 99% idle.

The Core Difference — At a Glance

GPU — Training
📐Operation: Matrix × Matrix
[batch × in] × [in × out] = [batch × out]
⚡Parallelism: Every output cell is independent → 10,000s of cores fire simultaneously
🔄Bottleneck: FLOPS (math) — cores are the limit, not memory
📦Batch = 32 samples: 32× more work, but same time — cores absorb it all
↩️Backprop: More matrix multiplications, all parallelizable — GPU keeps winning
CPU — Inference
📏Operation: Vector × Matrix
[1 × in] × [in × out] = [1 × out]
🐢Parallelism: Only "out" dot products — much less. GPU cores would mostly idle.
💾Bottleneck: Memory bandwidth — weights must be loaded for every single token
🔁Sequential: Token N+1 cannot start until token N is done. No way to batch time.
🏦RAM advantage: CPUs access hundreds of GB of cheap RAM — fits large models easily
