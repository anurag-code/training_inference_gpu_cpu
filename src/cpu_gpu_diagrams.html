CPU vs GPU

// why each processor fits a different deep learning task

Diagram 01

How Many Cores?

CPU

Few cores, each very powerful.
Built for complex sequential tasks.

GPU

Thousands of simple cores.
Built for doing one thing in parallel.

Key idea: A CPU core is like a genius who can solve any problem. A GPU core is like a factory worker who does one simple job — but you have 10,000 of them working simultaneously.

Diagram 02

What Happens at Inference?

STEP 1
Load weights
LOADING BILLIONS OF WEIGHTS FROM MEMORY…
Extremely expensive — gigabytes of data moved every token ⚠ BOTTLENECK
STEP 2
Pick 1 token
math
Tiny computation relative to the data loaded
STEP 3
Repeat…
LOAD ALL WEIGHTS AGAIN FOR NEXT TOKEN…
Sequential — cannot do token N+1 before token N
Conclusion: Inference is memory-bound, not compute-bound. You spend 95% of time moving data, 5% doing math. The GPU's massive parallel compute cores sit mostly idle — the CPU's sequential memory streaming is a better fit here.

Diagram 03

Memory Bandwidth — The Real Bottleneck

How fast can each processor read data from memory? (GB/s)

CPU
~80 GB/s
~50–100 GB/s
GPU
~3 TB/s (HBM)
~1–3 TB/s
Wait — GPU wins on bandwidth too? Yes, but remember: inference is sequential. Even with a wider pipe, you're still sending one truck at a time. The CPU's more moderate bandwidth is sufficient for sequential token generation, and CPUs can use cheap system RAM (large capacity, low cost) instead of expensive VRAM.

Diagram 05

At a Glance

Task Bound by Tokens processed Gradients? Best processor
Inference — generating text Memory bandwidth One at a time No CPU
Training — learning weights Compute (FLOPS) Entire batch Yes GPU
Bottom line: It's not about which processor is "better." It's about matching the processor to the nature of the work. Training floods every core with dense parallel math — GPUs love this. Inference streams data sequentially to produce one token — CPUs handle this efficiently without wasting expensive GPU resources.
